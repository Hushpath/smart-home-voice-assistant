# AGENTS.md

## 项目背景

本目录是《软件工程》课程大作业“智能家居语音交互助手系统”的前端部分。

本系统采用前后端分离架构。后端由 FastAPI 实现，位于 `backend/` 目录。前端使用 Vue 3 + Vite 实现 Web 原型界面，负责用户登录、Dashboard 展示、设备状态展示、真实语音输入、中文指令执行、操作日志展示、提醒管理、场景模式执行等功能。

前端通过 REST API 调用后端。后端是业务逻辑和数据结构的事实来源，前端应优先适配现有后端接口、schemas、routers、README 和 Swagger 文档。

## 与后端设计的连贯性要求

前端开发必须与 `backend/` 中已经实现的后端保持连贯。

在开发或修改前端接口调用前，应优先检查以下后端文件或信息：

1. `backend/README.md`
2. `backend/app/routers/`
3. `backend/app/schemas/`
4. `backend/app/models/`
5. 后端 Swagger/OpenAPI 文档 `/docs`

前端不得凭空假设接口字段。尤其需要确认：

1. 登录接口的请求体字段名称；
2. 登录接口返回的 token 字段名称；
3. `/api/commands/execute` 的请求体字段名称；
4. 设备对象中的字段名称；
5. 日志对象中的字段名称；
6. 提醒对象中的字段名称；
7. 场景对象中的字段名称；
8. 设备历史接口返回的数据结构；
9. 后端统一响应格式的实际结构。

如果后端实际字段与前端开发说明中的示例字段不同，必须优先适配后端实际字段。

前端可以在 `src/api/` 或 `src/utils/` 中编写数据适配函数，用于把后端返回数据转换为页面展示所需结构。

建议提供或完善以下适配函数：

1. `normalizeLoginResponse()`
2. `normalizeDevice()`
3. `normalizeDeviceHistory()`
4. `normalizeCommandResult()`
5. `normalizeCommandLog()`
6. `normalizeReminder()`
7. `normalizeScene()`
8. `normalizeWeather()`

## 后端修改原则

默认优先适配现有 `backend/`。

如果为了前后端联调确实需要修改后端，可以由 Codex 自主判断并进行最小范围修复，但必须保证不破坏既有接口、数据结构、测试和文档一致性。

允许修改后端的情况：

1. CORS 配置缺失；
2. 接口未正确注册到 FastAPI；
3. README 与实际启动命令明显不一致；
4. 前后端联调中发现明显 bug；
5. 后端已有接口返回字段存在小范围不一致，需要修复以符合既定设计；
6. 缺少必要错误处理，导致前端无法稳定演示。

禁止修改后端的情况：

1. 不要重构后端目录结构；
2. 不要随意修改已有接口路径；
3. 不要随意修改统一返回格式；
4. 不要重新设计数据库模型；
5. 不要删除鉴权逻辑；
6. 不要删除已有测试；
7. 不要为了前端方便伪造后端数据；
8. 不要引入新的大型后端依赖；
9. 不要接入真实智能家居硬件；
10. 不要接入外部大模型 API 或外部语音 API。

如果修改了 `backend/`，必须在完成报告中单独说明：

1. 为什么必须修改；
2. 修改了哪些文件；
3. 是否改变接口路径；
4. 是否改变响应字段；
5. 是否影响数据库；
6. 是否需要重新初始化数据库；
7. 是否需要重新运行后端测试。

## UI 设计要求

如果当前 Codex 环境提供 front design skill 或类似前端设计能力，可以让它根据“智能家居语音交互助手系统”的主题设计统一界面风格，并优化页面布局。

设计目标：

1. 页面要专业、现代、美观，适合课程演示、截图和录屏；
2. 整体应体现智能家居、语音交互、设备控制的主题；
3. Dashboard、设备卡片、语音控制面板、日志表格、提醒页面、场景卡片应保持统一视觉风格；
4. 信息层次清晰，核心操作突出；
5. 交互反馈明确，包括加载中、成功、失败、后端未启动、语音识别不可用等状态；
6. 桌面端优先，兼顾基础响应式；
7. 不要为了视觉效果引入复杂 3D、外部素材、大量动画或额外外部服务；
8. 不得改变既定功能范围、业务逻辑、后端接口和技术栈约束。

可以参考的设计方向：

* 智能家居控制台；
* 语音助手控制面板；
* 卡片式设备管理；
* 清晰的后台式导航布局；
* 现代、干净、统一的色彩和间距；
* 适合演示截图和录屏的视觉层次。

不要强制固定具体颜色、字号、圆角、阴影等细节，允许设计 skill 在上述边界内优化视觉效果。

如果项目中已经形成了某套统一视觉风格，后续页面应延续该风格，不要大幅改成另一套风格。

## 响应式要求

本项目以桌面端课堂演示为主，优先保证 1366×768、1920×1080 等常见桌面分辨率下显示正常。

前端需要做基础响应式适配：在手机或窄屏宽度下页面不能明显错位，不能出现严重横向溢出，卡片和表格应尽量换行或压缩显示。

不要求实现 App 级移动端体验，不做 PWA，不单独开发移动端版本。

开发优先级：

1. 桌面端完整可演示；
2. 窄屏不严重错位；
3. 有时间再优化手机端体验。

## 后端接口约定

后端基础地址默认：

```text
http://127.0.0.1:8000
```

前端开发地址默认：

```text
http://127.0.0.1:5173
```

后端 API 基础地址需要集中配置，建议放在：

```text
src/config/api.js
```

或 `.env` 文件中。

不要在多个页面中硬编码后端地址。

开发环境可以使用 Vite `/api` proxy。若需要直接跨域访问后端，允许在 `backend/app/main.py` 中用 FastAPI `CORSMiddleware` 做最小 CORS 修复，允许：

```text
http://127.0.0.1:5173
http://localhost:5173
```

## 鉴权约定

除登录、注册、健康检查和天气接口外，其余业务接口默认需要 JWT token。

登录成功后保存 token。请求受保护接口时必须添加请求头：

```text
Authorization: Bearer <token>
```

如果 token 缺失或失效，应跳转回登录页，并显示提示信息。

## 后端核心接口

认证：

* `POST /api/auth/register`
* `POST /api/auth/login`
* `GET /api/auth/me`

Dashboard：

* `GET /api/dashboard`

房间与设备：

* `GET /api/rooms`
* `GET /api/devices`
* `GET /api/devices/{device_id}`
* `PATCH /api/devices/{device_id}/state`
* `GET /api/devices/{device_id}/history`

中文指令：

* `POST /api/commands/parse`
* `POST /api/commands/execute`
* `GET /api/commands/logs`

提醒：

* `GET /api/reminders`
* `POST /api/reminders`
* `PATCH /api/reminders/{reminder_id}`
* `DELETE /api/reminders/{reminder_id}`

场景：

* `GET /api/scenes`
* `POST /api/scenes/{scene_id}/run`

天气：

* `GET /api/weather?city=北京`

## 后端返回格式

后端一般返回：

```json
{
  "success": true,
  "code": "OK",
  "message": "操作成功",
  "data": {}
}
```

前端要统一处理：

1. `success === true`：展示成功信息并使用 `data`；
2. `success === false`：展示 `message`；
3. 请求失败或后端未启动：展示友好错误提示，不要只在控制台报错；
4. token 失效：清除本地 token 并跳转登录页。

如果后端实际返回格式与上述示例不同，以后端实际代码和 Swagger 为准，并在前端请求层做兼容处理。

## 页面功能要求

### 1. 登录页

* 支持输入用户名和密码；
* 调用 `/api/auth/login`；
* 登录成功后保存 JWT token；
* 登录失败时显示错误信息；
* 登录成功后跳转到 Dashboard；
* 可提供默认账号提示，方便演示；
* 登录请求字段和 token 解析必须以后端实际 schema 为准。

### 2. Dashboard 首页

展示：

* 房间数量；
* 设备总数；
* 在线设备数；
* 当前开启设备数；
* 当前天气；
* 最近执行的指令；
* 常用语音指令示例；
* 快捷场景入口。

### 3. 设备控制页

* 按房间展示设备；
* 每个设备用卡片展示；
* 显示设备名称、房间、类型、在线状态、开关状态和关键参数；
* 支持手动打开/关闭设备；
* 支持展示空调温度、灯光亮度、电视音量、窗帘状态等属性；
* 操作后刷新设备状态；
* 支持查看设备状态历史；
* 设备展示字段必须通过后端实际 Device schema 适配。

### 4. 语音控制页

必须实现真实语音输入：

* 使用浏览器 Web Speech API；
* 点击按钮开始语音识别；
* 展示识别中的状态；
* 展示识别出的文本；
* 将识别文本发送到 `POST /api/commands/execute`；
* 展示后端返回的执行结果；
* 指令执行后刷新设备状态；
* 保留文本输入框作为语音识别失败时的兜底方式；
* 可选使用浏览器语音合成播报执行结果。

必须支持演示以下指令：

* 打开客厅灯
* 关闭卧室空调
* 把卧室空调调到26度
* 将客厅灯亮度调到80
* 把电视音量调到30
* 开启睡眠模式
* 提醒我晚上八点吃药
* 查询北京天气

`/api/commands/execute` 的请求体字段和返回结构必须以后端实际 schema 为准。

### 5. 操作日志页

* 调用 `/api/commands/logs`；
* 展示指令文本、执行结果、是否成功、执行时间；
* 使用表格展示；
* 支持刷新；
* 适合用于测试报告截图；
* 日志字段必须以后端实际返回结构为准。

### 6. 提醒管理页

* 调用 `/api/reminders`；
* 展示提醒列表；
* 支持新建提醒；
* 支持修改提醒状态；
* 支持删除提醒；
* 能展示通过语音创建的提醒；
* 提醒字段必须以后端实际返回结构为准。

提醒模块只需要前端展示和管理提醒事项，不需要实现真实系统通知。

### 7. 场景模式页

* 调用 `/api/scenes`；
* 展示回家模式、睡眠模式、离家模式；
* 每个场景用卡片展示；
* 点击按钮执行场景；
* 执行后刷新设备状态；
* 显示执行结果；
* 场景字段必须以后端实际返回结构为准。
